# Architecture gap fixes: review issue remediation (v2)

This commit addresses all five issues raised in the v1 review. The audit-logs route is rewritten to use Drizzle's `sql.join()` for dynamic WHERE construction, eliminating the broken `$1`-placeholder approach. Rate limiting (100/hour, IP-based) is added to all four CRUD POST handlers. Pagination via `limit`/`offset` query params is added to all four GET list handlers with sensible bounds (default 50, max 200). The missing `service-categories/[id]` route is created with GET/PATCH/DELETE following the price-books pattern exactly.

**Watch for:** Pagination responses omit a `total` count, making it impossible for clients to render page controls or know when they've exhausted results (confirmed). The rate-limit-before-auth ordering is inconsistent with the majority of existing routes but matches the import routes — functional but divergent (confirmed).

**Verdict**: APPROVED

## High-level view

The audit-logs rewrite uses `sql` template fragments joined with `sql.join(conditions, sql` AND `)`, which properly binds values through Drizzle's parameterization.

Rate limiting uses the existing `checkRateLimit` function (IP-based identification), so the before-auth placement works but diverges from the auth-first pattern in contacts/deals/leads/companies. Both orderings are present in the codebase already.

Pagination is bounded correctly (default 50, max 200) but the response body is `{ data: [...] }` without a `total` field. The audit-logs and contacts routes both return `{ data, total, limit, offset }` — the new tenant routes should follow that pattern for clients to paginate meaningfully.

## Pagination response shape is incomplete

The contacts route returns `{ data, total: countResult?.count ?? 0, offset, limit }`. The audit-logs route (fixed in this same task) also returns `{ data, total, limit, offset }`. The four new tenant list endpoints return only `{ data: results }` despite accepting `limit` and `offset` parameters.

A client calling `GET /api/tenant/products?limit=50&offset=0` gets 50 results but has no way to know if there are 51 or 5,000 total products. A "load more until empty" strategy works, but standard pagination controls (page numbers, "showing 1-50 of 347") cannot be rendered.

The fix is a single additional query per list endpoint:

```typescript
const [countResult] = await db.select({ count: sql<number>`count(*)::int` })
  .from(products)
  .where(and(...conditions));

return NextResponse.json({ data: results, total: countResult?.count ?? 0, limit, offset });
```

Not blocking — the endpoints are functional and bounded — but this API contract gap will surface as soon as any frontend tries to display pagination metadata.
