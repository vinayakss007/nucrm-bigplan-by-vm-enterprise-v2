# Architecture gap fixes: route canonicalization, CRUD scaffolding, FK constraints

This change resolves several documented architecture gaps from the project's full architecture map. Legacy `/api/super-admin/` routes now re-export from their canonical `/api/superadmin/` counterparts. Four new tenant CRUD resources (products, segments, service-categories, price-books) fill the gap between schema tables and exposed API surface. Missing foreign key constraints on five columns add referential integrity to billing, e-signature, hierarchy, and territory schemas. The invoice creation endpoint gains a uniqueness retry loop for generated invoice numbers.

**Watch for:** The superadmin audit-logs route builds a WHERE clause with `$1`-style placeholders then passes it through `sql.raw()` without binding the params array — query filters are non-functional (confirmed). The new CRUD routes omit rate limiting and pagination, diverging from existing tenant routes (confirmed). The service-categories resource has no `[id]` sub-route, so individual GET/PATCH/DELETE is not possible (confirmed).

## High-level view

The route canonicalization is clean: legacy files shrink to one-line re-exports from canonical paths, avoiding redirect latency or divergent logic. The only canonical route with new logic — `superadmin/audit-logs` — has a broken query-building approach that will silently return unfiltered results or fail at runtime.

Four new tenant CRUD resources follow the established pattern (requireAuth, validateBody, Drizzle ORM, soft-delete via `deletedAt`). They skip rate limiting on POST endpoints that every comparable existing route enforces, and return unbounded result sets without pagination support.

The FK constraint additions use appropriate cascade behaviors, but existing orphaned data in production could block the migration. The invoice uniqueness retry loop lacks transactional isolation, leaving a window for duplicate numbers under concurrent writes if no DB-level unique constraint exists.

The `service-categories` resource lacks an `[id]` sub-route, making it the only new resource that can be created and listed but not individually retrieved, updated, or deleted. The architecture map was marked RESOLVED for this item despite incomplete implementation.

## Audit-logs query construction is broken

The `superadmin/audit-logs` route builds a parameterized WHERE clause string (`WHERE admin_id = $1 AND action = $2 ...`) and accumulates values in a `params` array, then injects the string via `sql.raw(whereClause)`. Drizzle's tagged template `sql` function binds interpolated values at template positions — it has no mechanism to receive a separately-constructed params array. The `$1`, `$2` placeholders in the raw string are literal text from the database's perspective.

If any filter params are provided, the query will fail (Postgres sees `$1` without a bound parameter) or return an error. Without filters, the route works because `whereClause` is empty. The `LIMIT` and `OFFSET` are properly interpolated via the template literal, so the unfiltered path is fine.

The fix is to use Drizzle's `sql` helpers to build conditions dynamically:

```typescript
const conditions = [];
if (adminId) conditions.push(sql`admin_id = ${adminId}`);
// ...
const where = conditions.length ? sql`WHERE ${sql.join(conditions, sql` AND `)}` : sql``;
```

No security exploit path exists because the placeholders prevent injection even in their broken form — but no user of the audit log filtering will get correct results.

## Missing rate limiting on write endpoints

Every new POST handler (products, segments, price-books, service-categories) omits `checkRateLimit`. Existing comparable routes — contacts, deals, leads, companies — all enforce a per-hour cap (typically 100/hour). Authenticated users could create unbounded records in a burst.

## Unbounded list responses

All four new GET endpoints query with `orderBy(desc(...createdAt))` but no `LIMIT`. For tenants with thousands of products or segments, this returns the entire table. The existing `services/route.ts` also lacks pagination (same pattern was copied), so this is consistent with the template — but it's a latent scalability concern that compounds with each new resource.

## Incomplete service-categories resource

Products, segments, and price-books each have both a collection route (`route.ts`) and an item route (`[id]/route.ts`). Service-categories has only the collection route. A client can list and create categories but cannot fetch a single category by ID, update it, or soft-delete it. The architecture map section 6.2 marks this as RESOLVED with "full CRUD" — an overstated claim.

## Invoice uniqueness without transactional isolation

The count query and uniqueness check aren't wrapped in a transaction. Two concurrent requests could both pass the uniqueness check, then either one fails on insert (if `invoice_number` has a DB-level unique constraint) or both succeed with the same number (if it doesn't). The retry loop mitigates single-request races but doesn't close the concurrent-write window.

## FK constraints may block migration on orphaned data

The five new FK constraints (invoiceLineItems → invoices, orderLineItems → orders, signingRequests → documents, hierarchyPermissions → tenantHierarchy, territories → users) will cause the migration to fail if any existing rows reference IDs that don't exist in the parent table. A pre-migration data cleanup or `NOT VALID` constraint addition may be needed for production deployment.
