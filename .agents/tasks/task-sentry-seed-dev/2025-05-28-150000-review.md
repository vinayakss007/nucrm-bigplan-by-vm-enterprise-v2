# Development seed script for NuCRM

A ~1200-line TypeScript script (`scripts/seed-dev.ts`) that populates a local PostgreSQL database with realistic test data across 20+ tables: users, tenant, roles, companies, contacts, leads, pipelines, deals, tasks, tickets, activities, emails, forms, sequences, quotes, products, AI activity, modules, plugins, custom fields, API keys, webhooks, notifications, audit logs, automations, and plans. The script uses deterministic UUIDs so relationships are predictable, includes a production guard (`NODE_ENV !== 'production'`), and claims idempotency through a cleanup phase. Supporting changes add a `seed:dev` npm script and a DEV QUICK START section to `.env.example`.

**Watch for:** The cleanup phase only deletes 4 tables (tenantMembers, roles, tenants, users) and relies on `ON DELETE CASCADE` from the tenant FK to clean the rest, but two seeded tables (`exchange_rates`, `plans`) are not tenant-scoped and lack cleanup — `exchange_rates` will throw a duplicate key error on re-run (confirmed). The blanket `@ts-nocheck` suppresses all type errors in the file, masking schema mismatches that would surface as runtime failures against the actual database (confirmed).

## High-level view

The script achieves idempotency for tenant-scoped data by leveraging the schema's `ON DELETE CASCADE` on `tenantId` foreign keys: deleting the seed tenant cascades to companies, contacts, deals, and everything else scoped to it. This works for the 18+ tenant-scoped tables but leaves two global tables (`exchange_rates` and `plans`) to handle their own conflicts. `plans` uses `.onConflictDoNothing()` correctly; `exchange_rates` does not, breaking the idempotency promise on second run.

The `@ts-nocheck` directive at the top of the file is the script's biggest long-term liability. The schema has dozens of required fields, JSON column shapes, and enum constraints. Without type checking, any schema migration that renames or removes a column will not produce a compile error in this file — it will produce a runtime `INSERT` failure that a developer encounters only when they run `npm run seed:dev`. For a file explicitly designed to be re-run frequently, this is a high-maintenance risk.

The hardcoded password `password123` and webhook secrets (`whsec_dev_secret_123`) are reasonable for a dev-only script that refuses to run in production. The production guard checks `NODE_ENV` before any database operations. However, the guard is purely environment-variable based; nothing prevents a deploy pipeline from accidentally omitting `NODE_ENV=production` and running this script against a real database.

<summary>Issues (4)</summary>

1. **exchange_rates not idempotent** — The `exchangeRates` insert has no `onConflictDoNothing()` and no cleanup step. Second run will fail with a unique constraint violation on the currency pair index. Add `.onConflictDoNothing()` or delete by specific currencies in the cleanup phase.
2. **@ts-nocheck masks schema drift** — The entire file suppresses TypeScript errors. Remove `@ts-nocheck` and fix any type errors, or at minimum use targeted `@ts-expect-error` on lines where the schema genuinely doesn't match the insert shape (e.g., if optional fields are intentionally omitted). This ensures future schema changes surface as build errors rather than seed failures.
3. **Cleanup scope incomplete for re-run correctness** — The cleanup only explicitly deletes `tenantMembers`, `roles`, `tenants`, and `users`. If any table's FK to `tenants` doesn't cascade (or is added in the future without cascade), re-running will fail with FK violations on insert. Consider a more explicit cleanup that deletes from the same tables the script inserts into, in reverse dependency order.
4. **Production guard relies solely on NODE_ENV** — If a CI or deploy context fails to set `NODE_ENV=production`, this script could run against non-local databases. Adding a secondary check (e.g., rejecting `DATABASE_URL` values that don't contain `localhost` or `127.0.0.1`) would make the guard fail-safe rather than fail-open.

## Idempotency gap in global tables

The `exchangeRates` table is not tenant-scoped (no `tenantId` column — confirmed by reading the schema at `drizzle/schema/financial.ts`). The seed inserts two rows keyed by `(baseCurrency, targetCurrency)` with an index on that pair. On first run this works; on second run the insert attempts to create the same rows again. Unlike `plans` (which uses `.onConflictDoNothing()`) and `modules` (same), `exchangeRates` has no conflict handling:

```typescript
await db.insert(schema.exchangeRates).values([
  { baseCurrency: 'USD', targetCurrency: 'EUR', rate: '0.92150000', source: 'exchangerate-api' },
  { baseCurrency: 'USD', targetCurrency: 'GBP', rate: '0.79230000', source: 'exchangerate-api' },
]);
```

The fix is trivial: add `.onConflictDoNothing()` to match the pattern already used for `plans` and `modules`.

## Type safety and @ts-nocheck

The file starts with `// @ts-nocheck`. This means every `.insert().values()` call is entirely untyped — the compiler won't flag missing required columns, misspelled field names, or type mismatches between the schema and the seed data. Given that the script touches 20+ tables with complex schemas (JSONB columns, enums, nullable/not-null distinctions), this creates a class of bugs that only surface at runtime.

For a script that developers run as their first interaction with the codebase (`npm run seed:dev` is step 4 in the quick-start guide), a runtime crash during seeding is a poor onboarding experience. Removing `@ts-nocheck` and fixing any resulting type errors would make the seed script self-verifying against schema changes.

## Cleanup order and cascade dependency

The cleanup phase:
```typescript
await db.delete(schema.tenantMembers).where(sql`tenant_id = ${IDS.tenant}`);
await db.delete(schema.roles).where(sql`tenant_id = ${IDS.tenant}`);
await db.delete(schema.tenants).where(sql`id = ${IDS.tenant}`);
await db.delete(schema.users).where(sql`id IN (...)`);
```

The tenant deletion cascades to all tables using `utils.tenantId()` (which is `references(() => tenants.id, { onDelete: 'cascade' })`). This covers companies, contacts, deals, tasks, tickets, activities, emailLog, forms, sequences, quotes, products, webhooks, notifications, auditLogs, automations, aiActivity, customFieldDefs, customPlugins, and taxRates.

The `tenantMembers` and `roles` deletes before the tenant delete are necessary because `tenantMembers` references both `users` and `tenants`, and the script needs to delete users after the tenant. The ordering is correct for the current schema. The risk is that if a future table references `users` without cascade (or references a global table that this seed also inserts into), the cleanup will silently leave orphaned data.

## Production guard analysis

```typescript
if (process.env.NODE_ENV === 'production') {
  console.error('[ERROR] Cannot run seed script in production!');
  process.exit(1);
}
```

This guard only prevents execution when `NODE_ENV` is explicitly set to `'production'`. A staging environment with `NODE_ENV=staging` or a misconfigured CI with no `NODE_ENV` set would pass the guard. The `DATABASE_URL` check that follows only verifies the variable exists, not that it points to a local database. For a script that inserts test data with the password `password123` and hardcoded API keys, a secondary safeguard (hostname allowlist) would be prudent.
