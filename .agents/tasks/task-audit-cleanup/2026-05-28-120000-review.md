# Documentation and accessibility audit cleanup

This change updates four documentation files to reflect the already-completed v1 API removal and adds accessibility attributes to the plugin page's icon-only buttons. The UI fix removes a stale `/tenant/integrations` entry from the sidebar customization catalog and adds `role="switch"`, `aria-checked`, and `aria-label` to plugin toggle/expand/delete buttons.

**Watch for:** The migration guide now states v1 is removed but retains a "Deprecation Headers" section describing v1 response headers — a contradiction that could confuse consumers (confirmed). The timeline table compresses two distinct events into the same month without clear ordering (possible).

## High-level view

The documentation edits bring the migration guide, architecture doc, changelog, and gap analysis into alignment with the current state of the codebase — changing "deprecated" to "removed", adding a historical-reference banner, marking gap analysis item #8 resolved, and recording three missing changelog entries.

The accessibility additions on the plugin page add `role="switch"` with dynamic `aria-checked` and contextual labels to the toggle, plus `aria-label` on the expand/collapse and delete icon-only buttons.

## Migration doc internal contradiction

The "Deprecation Headers" section near the bottom of `API_MIGRATION_v1_to_v2.md` still reads: "All v1 responses **now include** deprecation headers" and lists example headers with `Sunset: Sat, 01 Dec 2026`. This is impossible if the routes are deleted — there are no v1 responses to carry headers. A developer reading the doc end-to-end hits a logical contradiction. Either delete the section, or rewrite it in past tense explaining what these headers looked like during the deprecation window (useful for anyone still debugging old client logs).

The migration checklist retains forward-looking language ("Remove v1 fallback code from your application"), which works as an action item for consumers, but the "Deprecation Headers" section's present tense implies the endpoints are still live.

## Accessibility scope gap vs. spec

FEAT-002 step 3 specified adding `aria-label` to the "Test Connection" and "Copy Webhook URL" buttons in the expanded plugin panel. These were not added. However, those buttons already have visible text content adjacent to the icon — the text serves as the accessible name, making explicit `aria-label` redundant. The implementation is correct; the task spec was over-specified for those particular buttons.
