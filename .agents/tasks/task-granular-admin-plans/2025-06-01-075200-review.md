# Granular super admin permissions and 4-tier plan system (v2)

A permission layer for super admin operations (17 permissions across 4 roles), canonical plan definitions (Free/Basic/Pro/Enterprise), deletion guards, and an access-control UI. The v1 review raised 5 issues; all have been addressed. The gate is now wired into AuthContext, the migration exists, the seed route checks `plans.manage`, self-demotion is blocked, and the type comment was added.

**Watch for:** Permission enforcement is partial — only 2 of the 7 new super admin routes call `requireSuperAdminPerm` (confirmed). The access-control GET route returns all admin data to any super admin including read-only roles without checking `users.view` (confirmed). The UI sends PATCH requests without a CSRF token header, which will be rejected by the CSRF middleware in production (likely).

**Verdict**: APPROVED

## High-level view

The permission gate is now live: `AuthContext.superAdminRole` is populated from the DB query, the gate reads it without an `any` cast, and two routes (`access-control PATCH`, `plans/seed POST`) enforce granular checks. The remaining routes (`tenants GET/POST/PATCH/DELETE`, `users GET/POST/DELETE`, `access-control GET`) still rely solely on `ctx.isSuperAdmin`. This means a `super_admin_readonly` user can still delete tenants (not owned by admins) and create users.

The self-demotion prevention blocks a user from changing their own role. A `super_admin_full` user can still demote *the last other* full-access admin, leaving only themselves — a different flavor of the same brittleness, not a blocker since the remaining admin retains full access and self-demotion is now blocked.

## Partial gate enforcement across new routes

The gate is called in exactly two places:

```ts
// access-control PATCH
const permCheck = requireSuperAdminPerm(ctx, 'users.manage');

// plans/seed POST
const permCheck = requireSuperAdminPerm(ctx, 'plans.manage');
```

The following routes have no granular check:
- `GET /api/superadmin/access-control` — returns all admin users (exposes email + role)
- `GET /api/superadmin/tenants` — returns all tenants with billing info
- `POST /api/superadmin/tenants` — creates tenants and users
- `PATCH /api/superadmin/tenants` — modifies tenant plans
- `DELETE /api/superadmin/tenants` — suspends or hard-deletes tenants
- `GET /api/superadmin/users` — returns all users
- `POST /api/superadmin/users` — creates users
- `DELETE /api/superadmin/users` — currently blocked, but still no perm check

A `super_admin_readonly` user retains full create/update/delete access to tenants. The gate infrastructure works correctly when called, but the rollout is incomplete.

## CSRF gap in client-side fetches

The access-control page sends:

```ts
fetch('/api/superadmin/access-control', {
  method: 'PATCH',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ userId, role: newRole }),
});
```

No `x-csrf-token` header. The billing page's seed button similarly omits it:

```ts
fetch('/api/superadmin/plans/seed', { method: 'POST' });
```

The route handlers don't call `requireCsrf` inline, so enforcement depends on whether Next.js edge middleware applies it globally. If global middleware enforces CSRF, these calls break with "CSRF token missing or invalid." If it's opt-in per route, they'll work but lack CSRF protection on state-changing admin operations — a fail-open posture on the most privileged endpoints in the system.
