# Granular super admin permissions and 4-tier plan system

A permission layer for super admin users (4 predefined roles with fine-grained capability checks), a canonical 4-tier plan definition (Free/Basic/Pro/Enterprise), deletion guards protecting super admin users and their tenants, enterprise plan enforcement, and a UI to manage admin roles. The approach is clean in concept — centralized definitions in `lib/`, gate function, API routes, and a React page — but the gate function is never wired into request handling, making the entire permission system inert at runtime.

**Watch for:** The `requireSuperAdminPerm` gate is dead code — `AuthContext` never carries `superAdminRole` and no route calls the gate (confirmed). The `super_admin_role` column has no migration file, so existing deployments will crash on any query touching it (confirmed). The seed endpoint is unprotected by granular permissions and can be called by any super admin including read-only roles (confirmed).

## High-level view

The permission model defines 17 permissions across 9 categories and maps them to 4 roles. The gate function (`requireSuperAdminPerm`) checks a `superAdminRole` field on the auth context, but the auth middleware never populates that field, so the gate always falls through to its `'super_admin_full'` default. No API route imports or calls the gate. The entire granular permission layer is declarative-only — it renders correctly in the UI matrix but has zero enforcement at the API level.

The schema adds `super_admin_role` to the users table but has no corresponding Drizzle migration SQL. On an existing database, the column won't exist, and any SELECT/INSERT referencing it will fail with a Postgres error. This is a deployment blocker.

The access control API's PATCH handler queries the requesting user's role from the database rather than trusting the context — a sound decision given the context gap. However, it doesn't prevent a `super_admin_full` user from demoting themselves, which could leave the system with zero full-access admins.

The enterprise plan enforcement lives only in the tenants PATCH handler. If a plan change happens through another path (Stripe webhook, migration script, direct DB), the lock isn't enforced — a constraint only as strong as the single code path it protects.

1. **Gate function is dead code** — `requireSuperAdminPerm` is defined but never imported by any API route. `AuthContext` doesn't include `superAdminRole`, so the gate always defaults to full access. Wire the gate into routes or remove it; currently it provides false confidence.

5. **`maxStorageGb` type mismatch in seed** — `plan-definitions.ts` declares `maxStorageGb` as `number` but the schema column is `numeric(6,2)`. The seed passes it through `.toString()`, which works, but the `PlanDefinition` interface should reflect the actual DB type or the seed should handle the conversion more explicitly to avoid future confusion.

## Permission gate wiring gap

`lib/permissions/super-admin-gate.ts` exports `requireSuperAdminPerm(ctx, permission)` which reads `(ctx as any).superAdminRole`. The cast to `any` bypasses type safety because `AuthContext` has no such field:

```ts
// lib/auth/middleware.ts
export interface AuthContext {
  userId: string;
  tenantId: string;
  roleSlug: string;
  permissions: Record<string, boolean>;
  isAdmin: boolean;
  isSuperAdmin: boolean;
  // ... no superAdminRole
};
```

The middleware builds the context from a JOIN of users and tenant_members, but never selects `super_admin_role` from the users table. The fallback in the gate — `const roleSlug = (ctx as any).superAdminRole || 'super_admin_full'` — means every super admin is treated as full-access regardless of what's stored in the database.

No route handler imports `requireSuperAdminPerm`. The new routes (`access-control`, `plans/seed`, `tenants`, `users`) all use the simple `if (!ctx.isSuperAdmin) return 403` pattern. The granular system is pure dead code.

## Missing database migration

The schema declares `superAdminRole: text('super_admin_role').default('super_admin_full')` but the migrations directory has no corresponding `ALTER TABLE users ADD COLUMN super_admin_role`. The access-control GET endpoint queries `users.superAdminRole`, which will produce a Postgres "column does not exist" error on any deployment that hasn't re-created the database from scratch.

## Seed endpoint authorization surface

```ts
// app/api/superadmin/plans/seed/route.ts
if (!ctx.isSuperAdmin) return NextResponse.json({ error: 'Super admin required' }, { status: 403 });
```

Any super admin can invoke this, overwriting plan limits and features for the entire platform. Once the granular permission system is wired, this should require `plans.manage`. In its current state, a read-only admin would bypass this check because the gate is never called.

## Self-demotion risk in access-control PATCH

The handler checks `requestorRole !== 'super_admin_full'` to reject non-full users from changing roles, but it doesn't check whether `userId === ctx.userId`. A sole full-access admin can demote themselves to `super_admin_readonly`, after which nobody has the privilege to promote anyone back. The fix is either preventing self-demotion or counting remaining `super_admin_full` users before allowing the change.

## Enterprise plan enforcement — single code path

The enforcement lives only in the tenants PATCH handler. If a plan change happens through another path (Stripe webhook handler, future plan migration script, or direct DB update), the enterprise lock isn't enforced. This is worth documenting as a known limitation for this iteration, and the enforcement should eventually move to a database-level constraint or a shared validation function called from all plan-mutation paths.
