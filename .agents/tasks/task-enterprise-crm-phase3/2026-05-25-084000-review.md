# Phase 3: Five Feature Groups â€” Compliance, Vertical Modules, Communications, Financial, Sales Intelligence

This phase adds ~9,800 lines across 85 files implementing five feature groups that round out the platform's module catalogue. Each group follows the established pattern: Drizzle schema, typed library with business logic, Next.js API routes with `requireAuth` + `requireModule` gating, and unit tests. The work is mechanically consistent, compiles cleanly, and passes 360 tests (107 new).

Watch for: **Chat widget and email tracking endpoints accept unauthenticated requests with no rate limiting or module verification** (confirmed). **Visitor tracking uses the raw API key as the tenantId** without resolving it, letting arbitrary callers write into any tenant's visitor table (confirmed). **Assignment-rules and SLA PUT routes spread caller-controlled fields directly into the DB update** without allowlisting columns (confirmed). **Leaderboards endpoint has no module gate** â€” it's the only authenticated route in this phase that omits it (confirmed).

## High-level view

The compliance group (GDPR export/deletion, SOC 2 reporting, retention policies) is well-structured with proper admin-only guards on mutation. The SOC 2 report generator is more of an evidence-collection template than a real audit tool â€” some controls hard-code "pass" without inspecting actual state â€” but as a tenant-facing self-assessment artifact this is serviceable.

The vertical modules (SLA, auto-assignment, documents, report builder) demonstrate the strongest consistency in this phase: schema with appropriate indexes, lib code with well-typed interfaces, and routes that validate inputs. The SLA/assignment PUT routes are the exception â€” they allow callers to set arbitrary columns via spread, which is an exploitable API surface.

The communication group adds SMS (Twilio integration), live chat, and email tracking. The design decision to make the chat widget and email tracking pixel public (no auth) is correct for their use cases, but neither endpoint verifies the tenant has the relevant module enabled. The SMS webhook validates Twilio signatures in production mode only, creating a development/production parity gap.

Financial/operations (currency, tax, e-signature) is the most polished group. The tax calculation engine handles compound rates and line-item exemptions cleanly. E-signature uses a proper adapter pattern for DocuSign/HelloSign/internal. Webhook signature validation for e-signatures checks for header presence but does not actually compute HMAC verification.

Sales intelligence (leaderboards, territories, hierarchy, visitor tracking) is functional but has the most security surface area. The leaderboards route lacks module gating. Visitor tracking trusts a raw `x-api-key` header as the tenant identifier without resolving it through the API key table.

## Public endpoint security posture

Four endpoints in this phase are intentionally public (no `requireAuth`): chat widget (`/api/tenant/chat/widget`), email tracking pixel (`/api/tenant/email/track`), visitor tracking (`/api/tenant/visitors/track`), SMS webhook (`/api/tenant/sms/webhook`), and e-signature webhook (`/api/tenant/esignature/webhook`).

The chat widget POST accepts a `tenantId` in the request body and calls `createChatSession` directly. It does not verify the tenant exists, does not check if the `service-helpdesk` module is enabled for that tenant, and does not rate-limit. An attacker can create unbounded chat sessions against any tenant UUID.

The visitor tracking endpoint uses the `x-api-key` header as the tenant identifier directly:

```typescript
// For now, use the key directly as tenantId (simplified)
const tenantId = apiKey;
```

This comment marks it as a temporary shortcut, but in production this means any arbitrary string becomes a valid tenant scope. Worse, the endpoint inserts a visitor record with an attacker-controlled `id` field (`body.visitorId`) â€” if a UUID collision occurs with an existing visitor in a different tenant, the upsert logic updates the wrong record (the `WHERE` filters on both `id` and `tenantId`, so this specific exploit doesn't cross tenants, but the raw `tenantId` trust is still the core issue).

The email tracking GET endpoint writes `emailOpens`/`emailClicks` records for any `tid` parameter without verifying the tenant ID exists or that the campaign/email IDs are legitimate. While tracking pixels are inherently public, the lack of any signed token means an attacker can spoof opens at scale to corrupt analytics data.

## Assignment and SLA update routes â€” unconstrained field spread

Both `PUT /api/tenant/assignment-rules` and `PUT /api/tenant/sla` follow this pattern:

```typescript
const { id, ...updates } = body;
const [updated] = await db.update(table)
  .set({ ...updates, updatedAt: new Date() })
  .where(and(eq(table.id, id), eq(table.tenantId, ctx.tenantId)));
```

The `.set()` call passes all remaining body fields directly to the database. A client can include `tenantId`, `createdAt`, `deletedAt`, or any other column in the request body and overwrite them. The tenant-scoped WHERE clause prevents cross-tenant writes, but within a tenant this allows privilege escalation (e.g., setting `isActive` or `priority` on assignment rules without being explicitly validated).

The retention policy PUT in the compliance group does this correctly â€” it builds an explicit `updateData` object and only copies known fields.

## Leaderboards â€” missing module gate

The leaderboards route at `/api/tenant/leaderboards` imports `requireAuth` but never calls `requireModule`. Every other authenticated endpoint in this phase gates on the appropriate module. The leaderboards logic itself queries `deals` and `activities` tables, so any tenant can access it regardless of plan. This should gate on `analytics-pro` to be consistent with the report builder.

## E-signature webhook HMAC gap

The DocuSign adapter's `validateWebhook` implementation:

```typescript
validateWebhook(_payload: unknown, headers: Record<string, string>): boolean {
  const signature = headers['x-docusign-signature-1'];
  return !!signature;
}
```

This checks header presence, not correctness. The HelloSign adapter has the same issue with `x-hellosign-event-hash`. The `InternalAdapter` returns `true` unconditionally, which is acceptable for the internal provider. For external providers, this means any caller who includes the expected header (even with a garbage value) will pass validation.

## SMS webhook â€” dev/prod signature gap

The Twilio webhook validates signatures only when `NODE_ENV === 'production'`:

```typescript
if (process.env['NODE_ENV'] === 'production') {
  const signature = req.headers.get('x-twilio-signature') || '';
  if (!validateTwilioSignature(signature, url, params)) {
    return NextResponse.json({ error: 'Invalid signature' }, { status: 403 });
  }
}
```

In staging/preview environments (where `NODE_ENV` might be `development` or unset), signature validation is entirely skipped. Combined with the tenant resolution strategy (using `params['TenantId']` from the webhook payload, falling back to a `DEFAULT_TENANT_ID` env var), an attacker in non-production environments can inject arbitrary SMS records.

The `validateTwilioSignature` function itself uses proper HMAC-SHA1 with the auth token, so the production path is sound.

## Schema design observations

All 12 new schema files follow the same structural conventions: `utils.pk()` for UUID primary keys, `utils.tenantId()` for the foreign key, `utils.lifecycle()` for timestamp columns, and per-table indexes prefixed with `idx_`. The hierarchy schema is the only one that omits indexes â€” `tenant_hierarchy` and `hierarchy_permissions` have no explicit index definitions, which will hurt query performance for tenants with many hierarchy entries.

The `exchangeRates` table is tenant-agnostic (no `tenantId` column), shared across the platform. This is a reasonable design for exchange rates but differs from every other table in this phase.

The `visitors` table uses a caller-supplied `id` (from the tracking script) rather than generating a UUID server-side. This is intentional for client-side visitor fingerprinting, but means the ID format is not guaranteed to be UUID despite the schema column type.

## Report builder â€” raw SQL with string interpolation

The report builder route uses `db.execute(sql`...`)` with Drizzle's tagged template syntax, which parameterizes values correctly. The `custom_reports` table is referenced by raw table name (not imported from a schema file), suggesting the table isn't defined in the Drizzle schema and relies on an existing migration. This won't cause runtime errors if the table exists, but it bypasses type safety.

## Test coverage assessment

All 10 new library modules have corresponding unit test files. Tests cover the core business logic (template interpolation, SLA breach detection, territory tree building, tax calculation, visitor scoring) with mocked database calls. The mocking strategy is consistent: Vitest mocks of `@/drizzle/db` and schema modules.

Not tested: API route handlers themselves are not exercised in this phase. No integration tests verify `requireAuth` + `requireModule` gating. The public endpoints (chat widget, visitor tracking, email tracking, webhooks) have no tests verifying their security behavior under malformed or malicious input. The spread-update vulnerability in assignment/SLA PUT routes would only be caught by API-level tests that pass unexpected fields.
