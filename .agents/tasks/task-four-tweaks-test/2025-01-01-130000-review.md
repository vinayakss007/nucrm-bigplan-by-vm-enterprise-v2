# Sidebar trim, webhook wiring, and v1 API removal

This change does four things: hides the Integrations nav item from the sidebar, wires `fireWebhooks` calls into all CRUD routes (leads, deals, tasks, companies â€” both create and update/delete paths), deletes the entire deprecated `/api/v1/` directory along with `lib/api/deprecation.ts`, and removes the CSRF bypass that exempted v1 routes. The WebhookEvent type is extended with `task.deleted`, `company.deleted`, `lead.updated`, and `lead.deleted` to cover the new call sites.

Watch for: the `fireWebhooks` call in tasks POST passes `(newTask as any).id` but the activity log on the preceding line uses `(newTask as any)[0].id` â€” one of them is wrong, and the webhook call is the likely culprit (confirmed). The tasks PATCH handler never fires `task.completed` despite the event existing in the type and being subscribed to in the UI. The CSRF bypass removal for `/api/v1/` is safe only because the routes themselves are gone â€” but any client still hitting those paths will now get CSRF rejections rather than 404s depending on middleware ordering (likely).

## High-level view

The v1 API deletion is clean: nine route files and the deprecation helper are removed, and the one infrastructure reference (CSRF bypass) is cleaned up. No imports elsewhere reference these deleted files, so no dangling pointers.

The webhook wiring is mechanically consistent across routes â€” fire-and-forget with `.catch(() => {})` â€” but has two gaps. The tasks POST route accesses `(newTask as any).id` while the adjacent line treats the same variable as an array (`[0].id`), meaning the webhook payload likely contains `undefined` as the entity ID. And the tasks PATCH handler doesn't fire `task.completed` when a task is marked done, leaving a hole that the webhook settings UI and automation engine both advertise as available.

The sidebar change is a single-line removal of the Integrations item and its unused `Plug` icon import.

## Webhook ID mismatch in tasks POST

In `app/api/tenant/tasks/route.ts`, the insert returns an array via `.returning()`. The activity log correctly accesses element zero:

```typescript
entityId: (newTask as any)[0].id,
```

But the webhook call on the very next statement uses:

```typescript
fireWebhooks(ctx.tenantId, 'task.created', { id: (newTask as any).id, title: v.title }).catch(() => {});
```

Since `newTask` is an array, `.id` resolves to `undefined`. The webhook payload will carry `{ id: undefined, title: "..." }` â€” silent data corruption that won't surface until a consumer tries to use the ID. This is confirmed by reading both lines in sequence.

## Missing task.completed webhook in PATCH handler

The `task.completed` event is defined in the `WebhookEvent` type, appears in the webhook settings page event picker, is referenced in the automation engine's trigger types, and is documented in the SDK README. However, the PATCH handler in `app/api/tenant/tasks/[id]/route.ts` only imports `fireWebhooks` â€” it never calls it. When `body.completed === true`, the handler sets `status = 'completed'` and logs an audit event, but no webhook fires.

The `task.deleted` webhook was added to the DELETE handler in the same file, so this looks like an oversight rather than a deliberate omission.

## CSRF behavior after v1 removal

The removed line in `lib/auth/csrf.ts` was:

```typescript
if (path.startsWith('/api/v1/')) return false;
```

With the v1 routes deleted, any POST/PATCH/DELETE to `/api/v1/*` will now hit CSRF validation (returns true from `needsCsrfValidation`), then fail because there's no route handler to match. The failure mode depends on middleware ordering: if CSRF middleware runs first, stale clients get a 403 CSRF error; if route matching runs first, they get a 404. Either is acceptable, but the 403 path could confuse anyone debugging a stale integration that hasn't been updated.
