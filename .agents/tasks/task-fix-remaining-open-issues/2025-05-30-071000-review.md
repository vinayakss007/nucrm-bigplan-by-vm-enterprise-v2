# Subscriptions table rename, legacy tag deprecation, and leaderboard caching

Three architecture map issues are resolved in a single pass: the billing.ts `subscriptions` table is renamed to `service_subscriptions` at the DB level (with matching index renames) to eliminate a name collision with infra.ts; the legacy `contactTags` and `leadTags` tables in crm.ts receive `@deprecated` JSDoc annotations pointing developers to `entityTags`; and the leaderboards API wraps its raw SQL queries in `withCache` with a 300-second TTL. The architecture map section 6 is updated to mark these items RESOLVED.

**Watch for:** The leaderboard cache key includes `endDate.toISOString()`, but for non-custom periods the `endDate` is `new Date()` â€” meaning every request within the 5-minute TTL window still generates a unique cache key and the cache is never hit (confirmed). The table rename is a breaking schema migration with no accompanying migration file; a deploy without coordination will fail against existing data (likely).

## High-level view

The billing table rename changes the underlying DB table string from `'subscriptions'` to `'service_subscriptions'` while the TypeScript export name (`serviceSubscriptions`) stays unchanged â€” existing code continues to compile. The concern is purely operational: any production database still has the table named `subscriptions` and won't automatically become `service_subscriptions` without a migration.

The leaderboard caching wraps the entire query-and-rank pipeline in `withCache(key, fetcher, 300)`. The key incorporates tenant, metric, period, and date range. But `getDateRange` computes `endDate` as `new Date()` for all non-custom periods, so the ISO string in the cache key changes on every request (millisecond precision), defeating the cache entirely unless the caller provides explicit `start` and `end` params.

## Cache key instability on the leaderboard endpoint

The cache key is constructed as:

```typescript
const cacheKey = `leaderboard:${ctx.tenantId}:${metric}:${period}:${startDate.toISOString()}:${endDate.toISOString()}`;
```

For a `period=month` request, `startDate` is stable (first of the current month, midnight UTC) but `endDate` is `new Date()` â€” yielding values like `2025-05-30T07:10:00.123Z`, `2025-05-30T07:10:00.456Z`, etc. Each request produces a distinct key. The `withCache` call proceeds to the fetcher every time, stores the result under a key that will never be queried again, and the cache accumulates dead entries until they expire.

The fix is straightforward â€” for non-custom periods, either omit the endDate from the key (since it's implicitly "now" and the 5-minute TTL handles staleness) or truncate it to a 5-minute bucket:

```typescript
const keyEnd = period === 'custom' ? endDate.toISOString() : '';
const cacheKey = `leaderboard:${ctx.tenantId}:${metric}:${period}:${startDate.toISOString()}:${keyEnd}`;
```

The test file validates that `withCache` is called with the correct key pattern and TTL, but the mock calls the fetcher immediately â€” the tests pass regardless of whether the cache actually hits in production.

## Schema rename without migration

Changing `pgTable('subscriptions', ...)` to `pgTable('service_subscriptions', ...)` means all generated SQL will reference `service_subscriptions`. If the production database still has the table named `subscriptions`, every query against `serviceSubscriptions` will error with "relation service_subscriptions does not exist".

This project has no `drizzle/migrations/` directory visible in the diff, so it likely uses `drizzle-kit push` or generates migrations as part of deployment. If push-based, the rename will be applied as a DROP + CREATE (losing data) unless explicitly handled. A safe migration would be:

```sql
ALTER TABLE subscriptions RENAME TO service_subscriptions;
ALTER INDEX idx_subscriptions_contact RENAME TO idx_service_subscriptions_contact;
ALTER INDEX idx_subscriptions_company RENAME TO idx_service_subscriptions_company;
ALTER INDEX idx_subscriptions_status RENAME TO idx_service_subscriptions_status;
```
