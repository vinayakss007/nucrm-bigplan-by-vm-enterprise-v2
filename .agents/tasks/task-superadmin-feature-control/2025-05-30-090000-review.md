# Super admin per-tenant feature control and user directory

This change adds a granular feature override system that lets the platform super admin enable or disable specific features (Telegram, AI, WhatsApp, etc.) per tenant regardless of their plan tier. It introduces a `tenant_feature_overrides` table, hooks the override check into the existing module gate at the top of both `requireModule()` and `requireFeature()`, and builds the supporting backend (user-directory, feature-control, email-blast APIs) and frontend (user-directory page with search/export, feature-control toggle matrix). The approach layers an early-return override on top of the existing plan-gate so existing behaviour is preserved when no override exists.

**Watch for:** The `checkFeatureOverride` error path fails open — a database error means no gate enforcement (confirmed). The email-blast endpoint has no rate limiting, confirmation step, or audit trail, creating a one-click mass-email risk (confirmed). The search parameter in user-directory is interpolated into an `ilike` pattern without escaping SQL wildcards, enabling unintentional query amplification (confirmed).

---

## High-level view

The gate system gains an override layer: before checking plan-level access, `requireModule()` and `requireFeature()` both query `tenant_feature_overrides`. If an override row exists, it short-circuits to allow or deny. If the query errors, the override is silently skipped (fail-open). This means a transient DB issue doesn't block access, but it also means a revoked feature could leak through during outages.

The email-blast API provides an unbounded bulk-send capability. It queries the full target set then sends through `Promise.allSettled` in batches of 10. There's no confirmation flow, no recipient cap, no audit log entry. For a system where a single misfire sends to every user on the platform, the absence of guardrails is the primary concern.

The user-directory API exposes every user's email, phone, tenant memberships, and admin status across the entire platform through a single endpoint behind a super-admin check. Soft-deleted users are not filtered, appearing in results and eligible for email blasts.

The feature-control UI renders a toggle matrix of tenants vs features. Once an override is created, there's no path to delete it and return to plan-default — the toggle only cycles between enabled and disabled.

1. **Gate fails open on DB error** — `checkFeatureOverride` catches all errors and returns `{ found: false }`, which means the override is skipped and normal plan-gate runs. If the super admin *disabled* a feature via override and the DB is unreachable, access is granted. Return a 500 instead of silently skipping. (confirmed)
2. **Email blast has no rate limit or cap** — POST `/api/superadmin/email-blast` can send to the entire user base in one request with no confirmation, recipient cap, or cooldown. Add a per-request recipient limit (e.g., 500), a cooldown between blast calls, or a two-step confirm flow. (confirmed)
3. **Email blast writes no audit trail** — No audit log entry records who sent what to whom. A blast to all users is unaccountable. Insert into `auditLogs` with action, subject, target spec, and count. (confirmed)
4. **Search parameter enables wildcard amplification** — The `q` param is embedded in `%${q}%` without escaping `%` and `_` characters. A search string like `%%%%%` forces a full-table regex scan. Escape SQL wildcards before interpolation. (confirmed)

6. **No deletedAt filter on user-directory query** — The user-directory query joins `users` without filtering `isNull(users.deletedAt)`, meaning soft-deleted users appear in the directory and can receive email blasts. (confirmed)
7. **Override toggle UX has a three-state cycle gap** — The toggle button treats "no override" and "disabled override" identically on click (both enable). There's no way to remove an override and return to plan-default state once one is set. (confirmed)

## Gate override: fail-open on error

In `checkFeatureOverride` (gate.ts), the catch block returns `{ found: false }`:

```typescript
} catch (error) {
  console.error('[gate.checkFeatureOverride] Error:', error);
  return { found: false };
}
```

When `found` is `false`, both `requireModule` and `requireFeature` proceed to the existing plan-gate logic. For the "grant access" case, an error means the user falls through to normal plan checks — no worse than before. But for the "revoke access" case, an override with `enabled: false` was explicitly set by the super admin to *deny* a feature. During a DB error, that denial evaporates and the feature becomes accessible until the database recovers. The gate should return a 500 response when the override query fails, since failing open on an explicit deny is worse than temporarily blocking access. (confirmed)

## Email blast: unbounded send with no audit

The email-blast endpoint queries the full target audience and iterates through batches of 10 using `Promise.allSettled`. Missing safeguards:

- No maximum recipient count per request
- No cooldown between invocations
- No audit log entry recording the blast event
- No dry-run or confirmation flow
- `filter_verified_only` defaults to false, meaning unverified addresses receive blast emails unless explicitly opted out

A super admin misclick or rogue API call with `target: 'all'` sends to every platform user with no undo and no record beyond server logs. (confirmed)

## Search wildcard injection in user-directory

The search query is interpolated directly:

```typescript
ilike(users.email, `%${q}%`)
```

Drizzle parameterizes the value (preventing SQL injection), but `%` and `_` are LIKE metacharacters. A search for `%` matches every row regardless of table size. A search for `___@___` becomes a pattern match rather than a literal substring. Escaping these characters before embedding in the pattern ensures the search works as intended without triggering full-table scans on short patterns. (confirmed)

## Feature-key validation gap

The PATCH endpoint's zod schema validates `feature_key` as `z.string().min(1)` but does not constrain it to the keys defined in `CONTROLLABLE_FEATURES`. An override row with `feature_key: 'telegramm'` (typo) is silently accepted and stored but never matches any gate check because `requireModule` and `requireFeature` are called with different strings. Adding `.refine(key => CONTROLLABLE_FEATURES.some(f => f.key === key))` or using `z.enum()` prevents dead rows. (confirmed)

## Soft-deleted users in directory and blast targets

The user-directory query and email-blast queries join `users` without filtering on `isNull(users.deletedAt)`. The schema uses `utils.lifecycle()` which adds `deletedAt`, and the feature-control GET correctly filters `isNull(tenants.deletedAt)`. The user-facing queries should apply the same pattern — otherwise deleted accounts appear in the directory, are counted in pagination totals, and are valid targets for email blasts. (confirmed)

## Toggle UX: no path to "remove override"

The feature-control page toggle has three visual states: enabled (green), disabled (red), and no-override (grey). Clicking grey sends `enabled: true`. Clicking green sends `enabled: false`. There's no interaction to *delete* the override row and return to plan-default. The PATCH API only does upsert, never delete. Once an override exists, the super admin can only flip between force-enabled and force-disabled, never return to "use plan defaults." This needs either a DELETE endpoint or a special `enabled: null` value that removes the row. (confirmed)
