# Monthly AI token budget enforcement per tenant

This change adds plan-based monthly AI token quotas to the multi-tenant CRM. Each plan tier gets a hard monthly token budget (free 10K, basic 100K, pro 500K, enterprise unlimited). A new `ai_token_usage` table tracks consumption per billing period. The enforcement sits in the AI route as a second budget gate after the existing `checkTokenAndLimits` call. A super admin page and API allow viewing usage across all tenants and overriding limits per-tenant.

Watch for: Race condition between budget check and usage recording lets concurrent requests overshoot the limit (confirmed). The PATCH endpoint accepts arbitrary `tenant_id` values without verifying the tenant exists (confirmed). No automated tests cover the new budget logic (confirmed).

## High-level view

The budget system is a straightforward check-then-record pattern layered on top of the existing token-budget machinery. The `checkAiTokenBudget` function resolves the effective limit (super admin override first, then plan default), compares it to the current period's usage, and returns an allow/deny decision. The recording side upserts with a SQL increment. The two operations are not atomic, which means under concurrent AI calls the limit can be exceeded by one request's worth of tokens before the next check sees the updated count.

The super admin API provides three operations via a single PATCH handler: set a custom limit, reset usage to zero, and implicitly create usage rows for tenants that haven't used AI yet. The input validation is minimal — `tenant_id` is trusted without a lookup, meaning a malformed or non-existent ID silently creates orphan rows in `ai_token_usage`.

The schema design uses `bigint` with `mode: 'number'` for token counts, which works for the defined limits (up to 500K) but will silently lose precision if a tenant accumulates beyond `Number.MAX_SAFE_INTEGER`. Given the current plan caps this is cosmetic, but the column type choice creates an implicit ceiling on future increases.

<summary>Issues (5)</summary>

1. **TOCTOU on budget check** — `checkAiTokenBudget` reads usage and `recordAiTokenUsage` writes it in separate transactions. Under concurrent requests, multiple calls can pass the check before any records. Use a single atomic `UPDATE ... WHERE tokens_used + $new < limit RETURNING *` or a serializable transaction with `SELECT FOR UPDATE`.
2. **Unvalidated tenant_id in PATCH** — The super admin endpoint accepts any string as `tenant_id` without verifying the tenant exists. Add a tenant existence check or a DB-level foreign key constraint on the upsert.
3. **No test coverage** — Neither `checkAiTokenBudget` nor `recordAiTokenUsage` nor the super admin API have unit tests. At minimum, test the allow/deny logic, the increment upsert, and the override resolution.
4. **`bigint` mode: 'number' precision ceiling** — Token counts use JavaScript numbers, which lose integer precision above 2^53. This is safe at current plan caps but should be documented or use `mode: 'bigint'` if limits could grow.
5. **Duplicate `getCurrentBillingPeriod` function** — The same function is defined in both `check-budget.ts` and the super admin route. Extract to a shared utility to avoid drift.

## TOCTOU gap in budget enforcement

The AI route calls `checkAiTokenBudget(tenantId)` which reads `tokens_used`, then proceeds with the AI gateway call, and only afterwards calls `recordAiTokenUsage(tenantId, tokensUsed)`. Between the read and the write, any number of concurrent requests for the same tenant can also pass the check. In `check-budget.ts`:

```typescript
const allowed = used < effectiveLimit;
```

This comparison uses a potentially stale `used` value. The `recordAiTokenUsage` function does an atomic `INSERT ... ON CONFLICT DO UPDATE SET tokens_used = tokens_used + $n`, which is correct for incrementing, but the check and the increment are in separate transactions. A tenant near their limit can fire several parallel AI calls and all of them will pass the check before any increment is visible.

The fix is to combine check and record into a single atomic operation: attempt an `UPDATE ai_token_usage SET tokens_used = tokens_used + $estimated WHERE tokens_used + $estimated <= limit` and treat zero affected rows as "denied". This eliminates the window entirely. Alternatively, for simplicity, accept that the overshoot is bounded to one request's token count (typically a few thousand) and document it as acceptable slack.

Confidence: **confirmed** — the two operations are plainly separate in `route.ts` lines where `checkAiTokenBudget` and `recordAiTokenUsage` are called with the entire gateway call between them.

## Input validation gap in super admin PATCH

The PATCH handler destructures `tenant_id` from the body and uses it directly in an upsert:

```typescript
const { tenant_id, tokens_limit, action } = body;
if (!tenant_id) { return ... 400; }
// No verification that tenant_id refers to an actual tenant
await db.insert(aiTokenUsage).values({ tenantId: tenant_id, ... })
```

If a super admin mistypes a tenant ID or the frontend sends a stale ID for a deleted tenant, the operation succeeds and creates an orphan row. The schema has `tenantId: utils.tenantId()` which likely adds a FK constraint at the Drizzle level — if so, the DB would reject it with a 500 error rather than a helpful 400. Either way, an explicit existence check with a clear error message would be cleaner.

Confidence: **confirmed** — no tenant lookup exists between the `if (!tenant_id)` check and the DB write.

## Absence of automated tests

The task notes say "All 435 tests pass" but no new tests were added. The budget logic has multiple branches: unlimited plan, override vs plan default, exact-boundary allowed/denied, first-request-creates-row vs subsequent-increments. The super admin API has three code paths (GET, PATCH-reset, PATCH-set-limit). None have test coverage. Given that the existing test suite doesn't mock `checkAiTokenBudget`, changes to this module could break the AI route silently.

Confidence: **confirmed** — grep for `checkAiTokenBudget`, `recordAiTokenUsage`, and `ai-token-usage` in `tests/` returns zero results.

## Schema design: `bigint` with JavaScript number mode

The `tokensUsed` and `tokensLimit` columns are declared as `bigint('tokens_used', { mode: 'number' })`. Drizzle's `mode: 'number'` means the value is marshalled as a JS `number`, not a `BigInt`. JavaScript numbers lose integer precision above `2^53 - 1` (~9 quadrillion). The current max plan limit is 500K tokens/month, so precision loss is impossible in practice. But the choice of `bigint` column type signals an expectation of large values — if that expectation is serious, `mode: 'bigint'` would be more defensive. This is a low-priority observation, not a bug.

Confidence: **confirmed** — declared at `drizzle/schema/ai-token-usage.ts` line 14-15.

## Duplicated `getCurrentBillingPeriod` utility

Both `lib/ai/check-budget.ts` and `app/api/superadmin/ai-token-settings/route.ts` define their own `getCurrentBillingPeriod()` function with identical logic. If the billing period format ever needs to change (e.g., fiscal calendar offset), both must be updated. Extract to `lib/billing/period.ts` or similar.

Confidence: **confirmed** — two identical implementations visible in the diff.
