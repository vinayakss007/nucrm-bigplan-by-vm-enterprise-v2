# CSRF cookie delivery fix and sidebar typography refresh

This change fixes a silent CSRF failure caused by `headers.set` overwriting the session cookie when setting the CSRF cookie during login/signup, adds a client-side recovery path that fetches a fresh CSRF token when the cookie is missing, and increases sidebar font sizes and dark-mode contrast. The `headers.set` → `headers.append` fix is the critical line—without it, users could authenticate but every subsequent state-changing request would 403.

**Watch for:** The `/api/auth/csrf-token` endpoint has no rate limiting, so an authenticated attacker could hammer it to generate unlimited tokens (confirmed). The CSRF recovery flow in CsrfProvider has no error handling around the refresh fetch itself—a network failure there produces an unhandled rejection (confirmed).

## High-level view

The root cause was a one-method bug: `headers.set('Set-Cookie', ...)` replaces any previously-set Set-Cookie header, so the CSRF cookie erased the session cookie. Switching to `headers.append` lets both cookies coexist. A new `/api/auth/csrf-token` GET endpoint and a recovery branch in CsrfProvider handle the edge case where a user's CSRF cookie disappears mid-session—the provider detects the missing cookie, hits the endpoint to re-set it, reads the fresh value, and retries the original request.

The UI changes are Tailwind adjustments: nav items from 14px/medium to 15px/semibold, section headers 12px→13px, icons 16px→18px, heading scale bumped, dark-mode muted text lightness 62%→72%, and shortcut label opacity 40%→70%.

1. **No rate limit on `/api/auth/csrf-token`** — An authenticated user can call this endpoint in a tight loop with no throttling. Add the same `checkRateLimit` wrapper used in login/signup.
2. **Unhandled fetch failure in CSRF recovery** — If the `/api/auth/csrf-token` fetch rejects (network error, timeout), the promise chain in CsrfProvider propagates an unhandled rejection instead of falling through gracefully. Wrap the recovery `.then()` in a `.catch()` that proceeds with the original request.
3. **Shortcut hover opacity goes *down*** — The shortcut label base is now `/70` but its `group-hover` state is `/60`, meaning hovering a nav item makes the shortcut *less* visible. Swap or equalize these values.
4. **No tests for the CSRF recovery path** — The new auto-recovery logic in CsrfProvider (fetch fresh token, retry) has no unit or integration coverage.

## CSRF cookie delivery: the `set` → `append` fix

`setSessionCookie(token)` uses the Next.js cookies API which internally appends a Set-Cookie header. When `api-handlers.ts` then called `response.headers.set('Set-Cookie', ...)`, the `set` semantics replaced that session cookie header entirely. Users appeared to authenticate but the session was never persisted to the browser—every subsequent request hit the auth wall.

## Token recovery endpoint and CsrfProvider auto-refresh

The recovery path has two problems:

```typescript
return originalFetch('/api/auth/csrf-token', { credentials: 'same-origin' }).then(() => {
  const freshToken = getCsrfToken();
  if (freshToken) {
    return performFetchWithToken(originalFetch, input, init, freshToken);
  }
  return originalFetch(input, init);
});
```

First, there's no `.catch()`. If the refresh fetch fails (offline, 500, timeout), the returned promise rejects with whatever the network error was, and the caller's original request never fires. A `.catch(() => originalFetch(input, init))` would let the request proceed and fail naturally at the proxy's CSRF check rather than disappearing into an unhandled rejection.

Second, there's no deduplication. If multiple concurrent requests all find the cookie missing simultaneously, each will independently fire a GET to `/api/auth/csrf-token`. A simple module-level `let refreshPromise: Promise | null` would coalesce concurrent recovery calls into one.

## No rate limiting on the token endpoint

Login and signup both call `checkRateLimit(request, ...)` before doing work. This new endpoint has no such guard. An authenticated user (or a compromised session) could generate thousands of tokens per second. The risk is low (each token is stateless, no server-side storage) but it's inconsistent with the rate-limiting posture on all other auth endpoints.

## Shortcut label hover opacity inversion

```tsx
<span className="text-[10px] text-muted-foreground/70 group-hover:text-muted-foreground/60 transition-colors">
```

Base opacity is 70%; on hover it drops to 60%. Prior to this change the values were `/40` base, `/60` hover (opacity increasing on hover), which was directionally correct. The new base `/70` should pair with hover `/80` or stay flat at `/70`.

## Bundled unrelated fix

The `<T,>` trailing-comma fix in `app/tenant/contacts/[id]/page.tsx` resolves a TSX parser ambiguity unrelated to CSRF or sidebar styling. It was apparently a pre-existing build blocker.
