# Custom plugin system with HTTP execution engine

A complete plugin subsystem separate from the existing integrations layer. Users can define custom HTTP API connections with 6 auth types, multiple actions per plugin using `{{variable}}` interpolation, inbound webhook receiving with HMAC-SHA256 verification, and full execution logging. The implementation spans database schema, execution engine, CRUD API, webhook handler, management UI, and docker-compose separation for dev vs production.

**Watch for:** Credentials stored in plaintext in the `auth_config` JSONB column (confirmed). Webhook endpoint lacks rate limiting or replay protection (confirmed). The `verifyWebhookSignature` comparison uses `===` instead of constant-time comparison, enabling timing attacks against webhook secrets (confirmed). PATCH endpoint accepts arbitrary field updates without validating enum values for `authType` or `status` (confirmed).

## High-level view

The plugin schema introduces two tables (`custom_plugins`, `plugin_execution_logs`) using the project's existing Drizzle utility patterns, but `authConfig` stores credentials as plaintext JSONB â€” the column comment claims "encrypted references" but no encryption layer exists.

The execution engine resolves auth for all six types, interpolates templates, executes HTTP calls with a 10-second timeout, and logs results. On network failure, it automatically transitions the plugin to `status: 'error'`, which means a single transient timeout permanently marks the plugin degraded until someone manually re-enables it.

The webhook handler accepts inbound POSTs without tenant authentication (the plugin UUID acts as a bearer token). The signature verification has a timing side-channel and no protection against replay attacks.

The CRUD API follows the project's existing `requireAuth` pattern, but the PATCH endpoint passes `authType` and `status` values straight through without validating them against the allowed enum values, unlike the POST route which validates `authType`.

The UI is a single-page client component with tabbed navigation. The "Use Template" button switches to the create tab without pre-filling the form â€” the template feature is non-functional.

## Credential storage and the encryption gap

The `auth_config` column in `custom_plugins` is typed as `jsonb` with a comment claiming "stores token/username/password/client_id/etc encrypted references." No encryption happens anywhere in the pipeline. The `POST /api/tenant/plugins` route takes `authConfig` from the request body and writes it directly:

```typescript
authConfig: (body['authConfig'] as PluginAuthConfig | undefined) ?? { type: 'none' },
```

Bearer tokens, basic auth passwords, OAuth client secrets, and API keys all sit in plaintext in PostgreSQL. The project already has an `ENCRYPTION_KEY` environment variable referenced in the production docker-compose, so the infrastructure for envelope encryption likely exists â€” it's just not wired into this plugin system.

Beyond storage, the `GET /api/tenant/plugins/[id]` endpoint returns `authConfig` in the response body, which means credentials are sent back to the browser on every plugin detail view.

## Webhook signature timing attack

In `lib/plugins/webhook-handler.ts`:

```typescript
const expected = signature.startsWith('sha256=') ? signature.slice(7) : signature;
return computed === expected;
```

The comment says "constant-time comparison via string equality" but JavaScript's `===` on strings short-circuits on the first differing byte. Use `crypto.timingSafeEqual`:

```typescript
import { timingSafeEqual } from 'crypto';
return timingSafeEqual(Buffer.from(computed, 'hex'), Buffer.from(expected, 'hex'));
```

## Plugin error state machine

The engine's error path unconditionally marks the plugin as broken:

```typescript
await db.update(customPlugins)
  .set({ lastError: errorMessage, status: 'error' })
  .where(eq(customPlugins.id, plugin.id));
```

A DNS blip, a target server returning 503, or a client-side timeout (10s is aggressive for some APIs) all permanently degrade the plugin. There's no backoff, no failure counter, no distinction between transient and permanent errors. Users must manually re-enable the plugin after every intermittent failure. Consider only transitioning to `'error'` after N consecutive failures.

## PATCH endpoint validation asymmetry

The `POST` route validates `authType` against `VALID_AUTH_TYPES`:

```typescript
if (authType && !VALID_AUTH_TYPES.includes(authType)) {
  return badRequest(`Invalid authType. Must be one of: ${VALID_AUTH_TYPES.join(', ')}`);
}
```

The `PATCH` route skips this:

```typescript
if (body['authType'] !== undefined) updates['authType'] = body['authType'] as PluginAuthType;
if (body['status'] !== undefined) updates['status'] = body['status'];
```

Since `authType` and `status` are text columns (not Postgres enums), invalid values persist silently. The engine's `resolveAuth` switch would fall through with no match, causing auth to not be applied â€” the request goes out unauthenticated.

## Webhook endpoint as an unauthenticated write path

The webhook route skips `requireAuth` by design. The only gating is knowledge of the plugin UUID (v4, 122 bits of entropy) and optionally the HMAC signature. Without rate limiting, a leaked UUID allows unbounded writes to `plugin_execution_logs`. The route also logs failed signature verifications to the database, so even rejected webhooks consume storage.

## Template feature is a dead end

The "Use Template" button calls:

```typescript
onClick={() => { setTab('create'); }}
```

This navigates to the create form but doesn't pass any template data. The `CreatePluginForm` component has no mechanism to receive pre-filled values.
